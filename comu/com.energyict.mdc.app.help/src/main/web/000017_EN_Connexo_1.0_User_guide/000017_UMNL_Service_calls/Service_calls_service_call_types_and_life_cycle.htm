Service calls, service call types and life cycle

Understand service calls

What is a service call?

Service calls are used if responsibility for executing an action is passed to an external system like a billing system or an MDM. They allow you to track the status and progress of actions requested by third party systems through web services, api, importers, processes, etc. on different objects in Connexo such as devices and usage points.

Using service calls the operator can see what changed on an object and where the action originates from. All of this information enables the operator to investigate what is happening and to intervene if necessary.

Important

Service calls are a tracking mechanism and do not do any actions by themselves. The actual service is handled by the requester, i.e. the third party system. The service call functions as a tag.

Possible scenarios for a service call

These scenarios will mostly be executed by meter operators or data experts.

Troubleshooting

  • To investigate an unexpected configuration change on a device, view the service call actions from the corresponding device landing page and find out if:

  • an unexpected requester has done a configuration change or,

  • an expected requester has done an unexpected configuration change.

  • To investigate why a reading request on a device did not succeed, find the detail of the reading request action in the device’s service call overview and analyse its log data.

Track status

Every month an external system can request data for a number of devices for a given time frame. Connexo will check data availability and return data if available. If not, Connexo checks if the data is expected to come in. If yes, the service call state becomes WAITING. Connexo will check every day for service calls that are in WAITING mode if the corresponding data has arrived. If so, the data is returned to the external system.

Follow up actions on an object

An external system triggered an asynchronous configuration change on a device. View the service call actions overview from the corresponding device landing page and find out if the action has been executed.

Follow up bulk actions involving multiple objects

To investigate a disconnect on multiple devices, navigate to the relevant disconnect actions and verify progress of each of the disconnect actions.

Understand service call types

In Connexo third party system create service calls through the service call framework. This framework requires service call types to allow following up the incoming requests. Service calls are always associated with a service call type.

Each service call type has some mandatory properties such as a unique type name, a version and a log level. On top of that, each service call type has an associated service call type life cycle (See “Understand service call type life cycles”). The same service call type life cycle can be associated with multiple service call types. A service call is always in a certain state.

The properties of a service call are defined on a per-service call basis. For example, a service call created from a web service to disconnect a meter will need other properties than a service call created from a web service to collect meter data. The service call type links the service call handler to the correct custom property sets.

Service call types are installed by your service & delivery team and cannot be changed after creation.

Service call types can be viewed in Connexo Admin.

Understand service call type life cycles

A service call is always in a certain state, this is the current state. States are managed through a finite state machine. So, each service call is in only 1 state simultaneously. It can change from 1 state to another when initiated by a triggering event or a condition. This is called a transition. The service call type life cycle is defined by a list of its states and the triggering conditions for each transition.

The service call type life cycle is named and defined in the service call type which is installed by your service & delivery team. In order to limit existing life cycle diagrams, it is possible to have states and transitions removed from the life cycle but not added.

Important

The following restrictions to adjusting service call type life cycles apply.

  • The states Created, Pending, Ongoing, Successful and Failed are essential and must always be part of the subset of states.

  • There is always a path from the initial state (Created) to the end state (Successful).

The following states are available by default:

State

Description

CREATED

Initial default state.

REJECTED

For example, if the incoming web service could not be started because of errors, lacking privileges, etc. The actual rejection reason can be found in the service call log. This logging information can be found on the service call landing page. See “View service calls”.

PENDING

When the task has been cleared but has not yet started.

ONGOING

The web service has started and is currently active.

PAUSED

The web service has been started but is currently not progressing on the service request because it has been paused.

WAITING

The web service has been started but is currently not progressing because it is waiting on another process to finish.

SCHEDULED

The web service has been started but is currently not pogressing because it has been scheduled to run on a later date.

CANCELLED

The web service has been cancelled and no further progress is made by the web service.

SUCCESSFUL

The web service has successfully completed its task(s).

PARTIAL SUCCESS

The web service has completed all tasks but not all of them were successful. For example, in the case of a bulk action, some subtasks may have failed.

FAILED

The web service has finished but was unsuccessful in completing the task in its entirety.
