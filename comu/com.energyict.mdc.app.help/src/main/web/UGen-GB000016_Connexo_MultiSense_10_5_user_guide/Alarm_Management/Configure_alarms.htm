Configure alarm creation rules

Understand alarm creation rules

In MultiSense you can define when to raise an alarm by defining alarm creation rules. In this rule you can define which specific logbook events can raise an alarm as well as which events clear the alarm. If an alarm is cleared, you will still have to close it. You also have to option to select how many times an event should occur before an alarm is created.

In addition to the event type, you also have to configure which state the device should have before an event can create an alarm. It is for example not relevant to raise a tamper alarm when the device is still in stock.

There is also an option to increase or decrease the urgency of the alarm when the alarm is raised again or cleared. An alarm is cleared when an event occurs that cancels the event that raised the alarm. The alarm will get the status cleared, but it still needs to be closed. For example an alarm can be triggered by a tampering event caused by opening the meter. When the meter is closed, a clearance event is raised. This event will give the status cleared to the raised alarm. The alarm will remain to be open, allowing the operator to filter these alarms and check whether the event causing the alarm was really solved.

In the alarm creation rule you can select some actions to execute each time the alarm is raised or when it is overdue. You can choose to automatically assign it to a specific user and/or workgroup or close the alarm.

Add alarm creation rule for device events

This creation rule will allow you raise alarms when specific events occur on devices.

5

Steps

1.   Go to Administration > Alarm management > Alarm creation rules.

2.   Click Add rule and complete the following fields:

Parameter

Description

Name

Give a unique name.

Rule template

Select Device alarm template.

Raised on event type

1.   Define which device events will raise an alarm.

a. Click Add event types. Select Specify event type if you when to give in the event type code manually. This has the format x.x.x.x and you can use wild cards. For devices that do not have CIM event codes you can define the device code by inserting the code 0.0.0.0 in the Event type field.

b. Choose Specify event type parts if you want to automatically create the event type code by selecting the different parts of the code from a drop-down list. To activate the device code select NA from the list for every part of the event type.

2.   Click Add.

Cleared on event type

Define which device events can clear the alarm that was raised by the device events defined in the parameter Raised on event type.

To define the event type follow the steps described above.

Device state

Defines in which device state the event will raise an alarm. If the device is not in the selected state, then the event will not raise an alarm.

First select one ore more device types, then select one or more device states.

Raise events should

Select whether event creates a new alarm or has an effect on other existing open alarms.

When raised again

Only active if you select Log on existing open alarms.

You can select to do nothing or raise the urgency by one.

When cleared again

Only active if you select Log on existing open alarms.

You can select to do nothing or decrease the urgency by one.

Create alarms when at least

Define how many times an event should occur over a specified period of time before a raises an alarm.

The time span is defined by relative periods.

Alarm reason

What is the cause of the alarm. Type a custom reason or select one from the list.

Priority

Shows the default priority of the alarm, this is the sum of the Urgency and the Impact.

Priority levels are as followed:

o1 - 20: very low

o21 - 40: low

o41 - 60: medium

o61 - 80: high

o81 - 100: very high

Urgency

Define the urgency of the alarm, maximum allowed value is 50.

Impact

Define the impact of the alarm, maximum allowed value is 50.

Due date

When does the alarm need to be handled.

Comment

Add additional information to the alarm.

Actions

Define what needs to be done when the alarm is created. See “Add actions to alarm creation rules” for more information.

3.   Click Add.

Add actions to alarm creation rules

5

Steps

1.   Go to Administration > Alarm management > Issue creation rules.

2.   To add actions to a new creation rule: click Add rule.

To add actions to an existing creation rule: click the actions icon  and select Edit.

3.   Click Add action.

4.   Define when to perform the action: when the alarm was created or when the alarm has passed its due date.

5.   Select action: assign the alarm or start a process when the alarm is raised or close or assign the alarm when it is overdue.

6.   Click Add.

7.   To activate the rule click the actions icon  and select Activate.

Add a web service notification to handle the alarm in an external application

By adding a web service notification as an action on an alarm creation rule you can forward alarms to an external system for further tracking and handling. An example of such an external system is Jira (Atlassian). This requires that one or more web service endpoints have been created in Connexo Admin > Web services > Web service endpoints. We refer to the Connexo Admin user guide for more detailed information about web services.

1

Important

You can add only one web service notification per alarm creation rule.

5

Steps

1.   Go to Administration > Alarm management > Alarm creation rules.

2.   To add actions to a new creation rule, click Add rule.

To add actions to an existing creation rule, click the actions icon  and select Edit.

3.   Click Add action.

4.   Verify that At creation time is selected in When to perform.

Web service notifications can only be triggered at the moment when the alarm is created. It isn’t possible to use web service notifications on existing alarms that reach the overdue time.

5.   In Action select Web service notification.

The field Web service notification is added.

6.   In Web service notification, select the needed web service endpoint.

This drop down shows all web service endpoints that are available in Connexo. They have been created in Connexo Admin > Web services > Web service endpoints. Contact your Connexo administrator if you need additional options in this list and refer to the Connexo Admin user guide for more detailed information about web services.

7.   Select the checkbox Close alarm if the Connexo alarm should be closed after forwarding it to the external system for further treatment.

1

Important

If Close alarm is selected, the alarm will receive the status Forwarded after it is forwarded to the external service by the web service notification. Indeed, the alarm won’t be fixed in Connexo. If needed, the external system can post a comment on an alarm so that you can keep track of its status in this other system. Contact your Connexo delivery team for more information about the available options.

8.   Click Add.

The option Web service notification is now no longer available in the field Action.

9.   Verify that all other parameters for your alarm creation rule are correct and click Add (new rule) or Save (existing rule) to save your work.

(De)Activate alarm creation rules

Deactivating alarm creation rules allows you to temporarily inactivate rules without the need to remove them permanently.

5

Steps

1.   Go to Administration > Alarm management > Alarm creation rules.

2.   Click the actions icon  and select Deactivate/Activate.

Edit alarm creation rules

5

Steps

1.   Go to Administration > Alarm management > Alarm creation rules.

2.   Click the actions icon  and select Edit.

3.   Change the alarm creation rule parameters.

4.   Click Save.

Remove alarm creation rules

5

Steps

1.   Go to Administration > Alarm management > Alarm creation rules.

2.   Click the actions icon  and select Remove.

3.   Click Remove to confirm.
