Understand issue creation rules

MultiSense offers the possibility the configure issue creation rules. This option limits the time you spend managing issues because you create rules to define which events will automatically trigger an issue. You can also define when the issue needs to be solved as well as what needs to be done with the issue such as assigning it to a specific user.

Different parameters are required to complete the configuration of a creation rule. First you will have to indicate the type of issue, this can be for example a data collection issue. Next you need to select a rule template to define the parameters of the rule.

A rule template is a collection of parameters that are specific for a certain rule. The exact values of the parameters are defined in the rule itself. An example of such a template is the template to create an issue when a slope is detected. All parameters will be shown, but then you will have to complete the reading type, the trend period and the threshold.

 

Available rule templates

In MultiSense different rule templates are available by default, below you can find a description of these rule templates and when to use them:

Rule template

Description

Data collection issues

 

Events from meters of concentrators

This rule template is designed to create an issue when multiple events that are related occur.

Create issue when specific event occurs

Is a rule template designed to create issues for simple events such as a lost connection or a communication failure.

Data validation issues

 

Create issue when suspects cannot be estimated

An issue is created when the data cannot be estimated by the estimation rules used by the configured estimation task.

Add data collection issue creation rules

5

Steps

1.   Go to Administration > Issue management > Issue creation rules.

2.   Click Add rule. Complete the following fields:

Parameter

Description

Name

A unique name

Issue type

Select Data collection

Rule template

Select the template containing all parameters to configure the rule. Depending on the selected template additional parameters may be shown.

1.   Events from meters of concentrator: This creates an issue based on multiple events that are related.

  • Threshold (%): Indicates how many times the event has to occur before creating an issue. For instance, an issue is created when 50% of the concentrator’s meters can’t communicate.

2.   Create issue when specific event occurs.

  • Event: Event trigger the creation of an issue.

3.   Meter registration issues: Create an issue when a slave hasn’t been registered on a gateway for some time.

  • Delay (in hours): The amount of time that the concerned slave must be unregistered from a gateway before creating the issue.

Auto resolution

Close issues automatically when the connection or communication problem is solved.

This option is only available for specific rule templates.

When recurring

Select Increase urgency (+1) to automatically increase the urgency with 1 when the same issue is recurring.

This option is only available for specific rule templates.

Issue reason

What is the cause of the issue. Type a custom reason or select one of the predefined reasons.

This reason is used to group issues on the Issue overview page see “View issues overview”.

Priority

The priority of an issue is the sum of the urgency and the impact of the issue. Priority levels are as followed:

o1 - 20: very low

o21 - 40: low

o41 - 60: medium

o61 - 80: high

o81 - 100: very high

Urgency

Define the urgency of the issue, maximum allowed value is 50.

Impact

Define the impact of the issue, maximum allowed value is 50.

Due date

When does the issue need to be solved.

Comment

Add additional information to the issue.

Actions

Define what needs to be done when the issue is created. See “Add actions to issue creation rules” how to configure these actions.

3

Note

Depending on the selected rule template, additional fields will have to be completed such as thresholds and reading types.

Add actions to issue creation rules

5

Steps

1.   Go to Administration > Issue management > Issue creation rules.

2.   To add actions to a new creation rule: click Add rule.

To add actions to an existing creation rule: click the actions icon  and select Edit.

3.   Click Add action.

4.   Define when to perform the action: when the issue was created or when the issue has passed its due date.

5.   Select action: assign to a user or workgroup upon creation or assign or close the issue when it is overdue.

6.   Click Add.

Add a web service notification to handle the issue in an external application

By adding a web service notification as an action on an issue creation rule you can forward issues to an external system for further tracking and handling. An example of such an external system is Jira (Atlassian). This requires that one or more web service endpoints have been created in Connexo Admin > Web services > Web service endpoints. We refer to the Connexo Admin user guide for more detailed information about web services.

1

Important

You can add only one web service notification per issue creation rule.

5

Steps

1.   Go to Administration > Issue management > Issue creation rules.

2.   To add actions to a new creation rule, click Add rule.

To add actions to an existing creation rule, click the actions icon  and select Edit.

3.   Click Add action.

4.   Verify that At creation time is selected in When to perform.

Web service notifications can only be triggered at the moment when the issue is created. It isn’t possible to use web service notifications on existing issues that reach the overdue time.

5.   In Action select Web service notification.

The field Web service notification is added.

6.   In Web service notification, select the needed web service endpoint.

This drop down shows all web service endpoints that are available in Connexo. They have been created in Connexo Admin > Web services > Web service endpoints. Contact your Connexo administrator if you need additional options in this list and refer to the Connexo Admin user guide for more detailed information about web services.

7.   Select the checkbox Close issue if the Connexo issue should be closed after forwarding it to the external system for further treatment.

1

Important

If Close issue is selected, the issue will receive the status Forwarded after it is forwarded to the external service by the web service notification. Indeed, the issue won’t be fixed in Connexo. If needed, the external system can post a comment on an issue so that you can keep track of its status in this other system. Contact your Connexo delivery team for more information about the available options.

8.   Click Add.

The option Web service notification is now no longer available in the field Action.

9.   Verify that all other parameters for your issue creation rule are correct and click Add (new rule) or Save (existing rule) to save your work.

Edit issue creation rules

5

Steps

1.   Go to Administration > Issue management > Issue creation rules.

2.   Click the actions icon  and select Edit.

3.   Change the issue creation rule parameters.

4.   Click Save.

3.   Click Remove.
