Manage certificates

Using the trust store you can manage all general certificates that are used by different objects. The object specific certificates are managed in a different location in Connexo Admin.

You can add certificates or create a CSR (Certificate Signing Request) to later add the certificate.

View certificates

You can view and filter all the certificates that are available in Connexo.

5

Steps

1.   Go to Administration > Certificate management > Certificates.

2.   Enter criteria in the filter bar on top to narrow down the list of certificates. Don’t forget to click Apply to execute your filter.

3.   Select a certificate and view its details in the details pane below.

4.   Click the alias of a certificate in the table to open the certificate’s details page.

Understand CSR in Connexo Admin

When you do not have a certificate already available you can create a CSR in Connexo Admin. This request usually contains the public key for which the certificate should be issued, identifying information (such as domain name) and integrity protection (e.g. a digital signature).

In Connexo you have to specify the key attributes and CSR attributes in the same step. For certificate generation the offline method is used. This means that once you completed the parameters, you need to download the CSR and then send it the CA (Certificate Authority).

The CA will generate the certificate which you can download and then import into Connexo Admin. Generating a CSR will get the certificate status Requested. Once you import the certificate the status will change to Available.

Use the CSR importer

To support the preparation and shipment of devices with the certificates that are issued by the CA of the customer, an offline process is often needed because the manufacturer doesn’t have direct access to a customer CA.

For this reason, a Connexo system administrator wants to be able to import one or more CSRs (certificate signing requests) from a device manufacturer to use his own CA to request the certificates and then send them back to the manufacturer who can upload them to the device. In addition, the system administrator can also upload (in the same file) a number of trust anchors, i.e. certificates from a trust store, to include them on the device during manufacturing.

5

Steps

1.   Go to Data exchange > Upload file for import.

2.   In the drop-down, select CSR importer.

3.   Select the import file in File.

The import file can’t be larger than 100 Mb.

The import tool at the production facility expects a ZIP file of the following format:

  • Each device gets a subdirectory with as name the device serial number.

  • Each sub-directory contains the certificates associated with the device, with suiting prefixes:

  • The TLS server certificate has the name "tls-" + system title tls

  • The DLMS TLS server certificate has the name "dlms-tls-" + system title dlms-tls

  • The DLMS agreement server certificate has the name "dlms-agreement-" + system title

  • The DLMS signature server certificate has the name "dlms-signature-" + system title

  • The (optional) DLMS client certificates have the name "dlms-client-" + system title

  • The (optional) DLMS client trust anchors have the name "dlms-client-ca-" + system title

  • The (optional) HTTPS client certificates have the name "https-client-" + client_name

  • The (optional) HTTPS client trust anchors have the name "https-client-ca" + ca_name

  • The (optional) IEEE 802.1X trust anchor certificate has the name "dot1x-ca-" + ca_name

  • The certificates are returned in PEM format (base64 encoded DER with header and footer)

Request certificate from the certificate authority

You can request a certificate from a certificate authority using the CSR. For certificates with the status Requested (CSR), the Request certificate action is available.

5

Steps

1.   Go to Administration > Certificate management > Certificates.

2.   Click the actions icon next to the needed certificate and select Request certificate.

3.   In the popup, enter a timeout after which the request must give an error.

4.   Click Request to send the request to the certificate authority.

An acknowledgement is shown as soon as the certificate is received. When the certificate is received from the certificate authority, it is stored in the certificate store.

Import a certificate

When you receive a certificate from the CA after you sent a CSR, you can import this certificate to the CSR instance you created in Connexo Admin.

You can also import certificates to existing certificate instances. When there is already a certificate available, this will be overwritten by the newly imported certificate.

The certificate file format needs to be DER-encoded to be able to import it into Connexo.

5

Steps

1.   Go to Administration > Certificate management > Certificates.

2.   Click the actions icon next to the certificate instance and select Import certificate.

3.   Click Select file and select the certificate to upload.

4.   Click Import.

Add a CRL task

A CRL is a certificate revocation list and contains digital certificates that have been revoked by the issuing certificate authority (CA) before their scheduled expiration date. Therefore, they should no longer be trusted. This is the list of certificates that should be revoked.

In Connexo MultiSense you can create a recurring system task to request CRLs from a CA on a regular basis and then revoke the certificates that are on that list. Refer to the Connexo MultiSense user guide for more detailed information.

Revoke a certificate

You can revoke certificates that have been compromised or issued by mistake. Revoked certificates no longer appear in drop-down lists when managing user directories, security accessors and device security accessors. The revoke action is only visible for certificates that don’t have the status Requested or Revoked.

5

Steps

1.   Go to Administration > Certificate management > Certificates.

2.   Click the actions icon next to the certificate instance and select Revoke.

3

Note

  • If the certificate is used on a security accessor of a device, firmware version or import service, then the status of the certificate isn’t changed and it isn’t sent to the certificate authority.

  • If the certificate isn’t in use and the certificate authority is online, then a request for revoking the certificate is sent to the certification authority. You can specify a timeout for the request after which an error will occur. While the request to the certificate authority is in progress, a progress bar is shown. You will receive a confirmation when the certificate has been revoked. Its status is then changed into Revoked.

  • If the certificate isn’t in use and the certificate authority is offline, then a confirmation message asks you to send the certificate to the certificate authority that will change its revocation status. By clicking Revoke the status of the certificate in Connexo already changes into Revoked.

Revoke multiple certificates simultaneously

To save time you can revoke multiple certificates simultaneously in a bulk action. Revoked certificates no longer appear in drop-down lists when managing user directories, security accessors and device security accessors. Usually, certificates that have been compromised or issued by mistake are candidates to be revoked.

5

Steps

1.   Go to Administration > Certificate management > Certificates.

2.   Click Bulk action.

3.   In Select certificates, specify the certificates that will be revoked. Then, click Next.

4.   In Select action, select Revoke certificates. Then, click Next.

At this point all needed checks are done:

  • Only certificates that don’t have the status Requested or Revoked can be revoked.

  • Only certificates that aren’t used on a device, security accessor or user directory can be revoked.

 

5.   In Action details, enter a timeout for the request after which an error will occur. The timeout is set for each of the requests. Then, click Next.

6.   In Confirmation, you can overview what will be done during the selected action. Click Confirm to proceed.

3

Note

  • If the certificate is used on a security accessor of a device, firmware version or import service, then the status of the certificate isn’t changed and it isn’t sent to the certificate authority.

  • If the certificate isn’t in use and the certificate authority is offline, then a confirmation message asks you to send the certificate to the certificate authority that will change its revocation status. By clicking Revoke the status of the certificate in Connexo already changes into Revoked.

  • The certificates with status Revoked or Requested that can’t be revoked will be listed for your review.

7.   In the step Status you are able to monitor the progress of your action. After the progress bar indicates that the action has finished, you can see how many certificates have been revoked as well as a list of the certificates that couldn’t be revoked.

8.   Click Finish to end the bulk action.

Mark a certificate as obsolete

You can mark a certificate as obsolete to indicate that it is no longer used in the Connexo and if it shouldn’t be removed yet. Certificates with the status Obsolete can still be used on security accessors, firmware versions, import services and user directories.

5

Steps

1.   Go to Administration > Certificate management > Certificates.

2.   Click the actions icon next to the certificate that you want to mark as obsolete.

3.   Select Mark as obsolete.

The certificate status is now Obsolete.

4.   If needed, select Unmark obsolete from the actions icon  of an obsolete certificate to revert it back to its previous status.

3

Note

  • You can mark certificates that are still used in Connexo as obsolete. In that case, you will just have to confirm your action.

  • The action Mark as obsolete isn’t available for certificates with the status Revoked, Obsolete or Requested.

Tasks for renewing certificates and keys on devices

In order to facilitate the renewal of certificates and keys on devices two recurring tasks are configured in Connexo that,

  • Look for certificates and keys that will expire within the specified period.

  • Start a certificate or key renewal process for the found certificates and keys.

 

The following parameters are used to define how each task works.

Certificate renewal task

Parameter

Description

certificateProcessDefinitionId

The definition ID of the process that is triggered when the conditions are met. This is the certificate renewal process.

certificateDaysTillExpiration

This is the condition to trigger the process defined by certificateProcessDefinitionId. It is a particular number of days before the certificate actually expires.

Key renewal task

Parameter

Description

keyProcessDefinitionId

The definition ID of the process that is triggered when the conditions are met. This is the single key renewal process.

keyDaysTillExpiration

This is the condition to trigger the process defined by keyProcessDefinitionId. It is a particular number of days before the key actually expires.

The tasks are scheduled to run at least once a day and all devices are considered.

When the task is triggered, the name of the security accessor that is about to expire is passed to the process.

These tasks can be seen in task management. See “Consolidated system task management”.

1

Important

The process isn’t triggered for an expiring key if:

  • A process with the same keyProcessDefinitionId and accessorType is already running on a device.

  • The security accessor is incomplete or not used by at least one security set on a device.

  • The device isn’t in an operational stage. See “Device life cycle management” in Connexo MultiSense.
