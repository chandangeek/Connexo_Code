Service calls, service call types and life cycle

Understand service calls

What is a service call?

Service calls are used if responsibility for executing an action is passed to an external system like a billing system or an MDM. They allow you to track the status and progress of actions requested by third party systems through web services, api, importers, processes, etc. on different objects in Connexo such as devices and usage points.

Using service calls the operator can see what changed on an object and where the action originates from. All of this information enables the operator to investigate what is happening and to intervene if necessary.

Important

Service calls are a tracking mechanism and do not do any actions by themselves. The actual service is handled by the requester, i.e. the third party system. The service call functions as a tag.

Understand service call types

In Connexo third party system create service calls through the service call framework. This framework requires service call types to allow following up the incoming requests. Service calls are always associated with a service call type.

Each service call type has some mandatory properties such as a unique type name, a version and a log level. On top of that, each service call type has an associated service call type life cycle (See “Understand service call type life cycles”). The same service call type life cycle can be associated with multiple service call types. A service call is always in a certain state.

The properties of a service call are defined on a per-service call basis. For example, a service call created from a web service to disconnect a meter will need other properties than a service call created from a web service to collect meter data. The service call type links the service call handler to the correct custom property sets.

Service call types are installed by your service & delivery team and cannot be changed after creation.

Service call types can be viewed in Connexo Admin.

Understand service call type life cycles

A service call is always in a certain state, this is the current state. States are managed through a finite state machine. So, each service call is in only 1 state simultaneously. It can change from 1 state to another when initiated by a triggering event or a condition. This is called a transition. The service call type life cycle is defined by a list of its states and the triggering conditions for each transition.

The service call type life cycle is named and defined in the service call type which is installed by your service & delivery team. In order to limit existing life cycle diagrams, it is possible to have states and transitions removed from the life cycle but not added.
