Understand the out-of-the-box processes in Connexo Insight

These processes are available in Connexo Insight  by default. The standard out-of-the-box processes are marked by STD between square brackets at the end of the process name. For instance Install meter [STD]. Custom processes are described in the specific project documentation.

Install meter (out-of-the-box process)

This process handles the meter installation.

5

Steps

1.   Start the Install meter process on a usage point.

The parameters that are specific for this purpose appear on the Start process page.

Parameter

Description

New meter name

Name of the installed meter. This name will be used in reports and references.

Installation date

Date, hour and minutes when the meter must be installed.

Meter role

The list of available meter roles from all the metrology configurations after the installation date. Meter roles that already have linked meters are excluded from this list. If there is no metrology configuration after installation date, then all meter roles in Connexo are available in the list.

2.   Click Start to start the process.

 

What happens when the process is running?

If the installation date is in the past, then the following steps are executed.

5

Steps

1.   Connexo executes the process.

Connexo tries to attach the new meter to the usage point.

If no name was entered for the new meter, then a user task is created asking for a name for the newly installed meter.

If the meter isn’t compatible with the metrology configuration requirements because it doesn’t provide reading types for the purposes of the metrology configuration on the usage point or if the new meter isn’t found in Connexo, then a user task is created asking to provide another meter or to adjust the usage point or the meter accordingly.

2.   User completes the eventual user task.

3.   Connexo completes the process:

  • The new meter will be linked to the usage point starting from the configured installation date.

  • The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

 

If the installation date is in the future, then the following steps are executed.

5

Steps

1.   Connexo executes the process.

A user task is created asking to define the time slot for the installation of the new meter on the usage point.

2.   User completes the Actual installation date.

The actual installation date is taken into account by the process instead of the installation date that was completed when starting the process.

3.   The subprocess Install meter (physically) starts by creating a new user task asking to physically install the new meter on the usage point on the configured actual installation date and to provide the new meter’s serial number, device type and year of certification.

4.   The user installs the meter and completes the needed attributes.

5.   Connexo verifies if a meter with the configured name or serial number is installed in Connexo. This search is done first by name and then by serial number.

If the new meter isn’t installed in Connexo, then Connexo tries to create the device. The device configuration of the device type is based on the requirements of the actual metrology configuration on the installation date.

If the appropriate device configuration or device type isn’t found or if multiple device configurations were found, then a new user task is created asking to create the meter manually. The user is required to enter the new meter name in the user task.

6.   Connexo checks if the meter can be linked to the usage point at the actual installation date.

If the meter can’t be linked because it doesn’t provide reading types for purposes of the metrology configuration on the usage point, then a new user task is created asking to provide new meter parameters. The user completes a new serial number, device type and year of certification in the user task.

7.   Connexo executes the process on the new device and waits for its completion.

8.   Connexo checks if the device is in an operational stage and fetches the transition date to the operational stage for the new device.

If the device isn’t in an operational stage, then Connexo executes the process again. See step 7.

9.   Connexo links the new meter to the usage point using the activation date of the new meter. This date was fetched in step 8.

If the meter couldn’t be linked due to particular errors, then the process is aborted and the appropriate message is shown in the process execution log.

10.   Connexo completes the process:

  • The new meter will be linked to the usage point starting from the configured installation date.

  • The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

 

1

Important

The detailed process diagram can be seen in Connexo Flow. If you don’t have sufficient privileges due to a technical or commercial reason, find out who does or contact the Connexo service desk.

Interrupt service (out-of-the-box process)

Trigger the Interrupt service process on a usage point to stop providing services on that usage point because of billing, technical, security or other issues.

5

Steps

1.   Start the Interrupt service process on a usage point.

The parameters that are specific for this process appear on the Start process page.

Parameter

Description

Reason

Reasons related to electricity can be a lightning strike or a potential overload of the grid. Thermal, gas and water related reasons can be a leak or a pipe burst. Other reasons may be problems on site, security reasons or bad debt.

Trigger date

Date and time when the process is actually triggered.

2.   Click Start to start the process.

 

What happens when the process is running?

5

Steps

1.   Connexo executes the process.

If the trigger date is in the past, then the usage point connection status is changed from Connected into Logically disconnected. The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

If the trigger date is in the future and the service interruption isn’t allowed, then Connexo stops the flow. Status becomes Aborted and a text variable in Status overview under the History tab indicates what has happened. The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

If the trigger date is in the future and the service interruption is allowed, then Connexo triggers the target head end system (HES) to execute the command to interrupt the service.

3

Note

The following checks are done to see if the service can be interrupted.

  • Service category must be gas or electricity.

  • The attribute Interruptible must be checked.

  • Target date must be within a range that is defined in the process.

 

2.   Connexo triggers the target HES to execute the command to interrupt the service.

If there is no response after some pre-defined period, then Connexo stops the flow. Status becomes Aborted and a text variable in Status overview under the History tab indicates what has happened. The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

If the process was triggered on a usage point with an activated meter that is managed by a HES that isn’t supported, then a new user task is assigned to the data operator. This task contains a message that asks the data operator to contact another operator from the concerned HES to interrupt the service for the current usage point on the specified device on the defined trigger date. As an optional parameter an actual date can be added to this user task. By default, this date is set to the trigger date.

If the process is triggered on a usage point with an activated meter that is managed by a supported HES like Connexo MultiSense , then the HES executes the interruption command.

3.   The HES executes the interruption command.

If the interruption command executes successfully, then the usage point connection status is changed from Connected into Logically disconnected. The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

If the interruption command fails, then Connexo stops the flow. Status becomes Aborted and a text variable in Status overview under the History tab indicates what has happened. The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

4.   Connection state of the usage point has been changed to Logically disconnected.

Limit load (non smart) (out-of-the-box process)

Trigger the Limit load (non smart) process to plan the setting of the physical load limiter on non-smart meters that can’t be managed remotely. In this process, Connexo will only update the corresponding usage point attributes.

5

Steps

1.   Start the Limit load (non smart) process on a usage point.

The parameters that are specific for this process appear on the Start process page.

Parameter

Description

Trigger date

Date and time when the process is actually triggered.

Load limit

Quantity for the load limit.

Limiter type

Indicates how the load limit is achieved.

  • Control.

  • Fixed.

  • Variable.

  • No limiter.

Reason

Reasons related to electricity can be a lightning strike or a potential overload of the grid. Thermal, gas and water related reasons can be a leak or a pipe burst. Other reasons may be problems on site, security reasons or bad debt.

2.   Click Start to start the process.

 

What happens when the process is running?

5

Steps

1.   Connexo executes the process.

If the trigger date is in the future, then a new user task is created and assigned to the data operator. This task contains a message that asks to contact a field engineer to limit the load on the current usage point on the defined trigger date by means of the defined limiter type to the specified load limit quantity. If load limit value was set to 0 in the process parameters, then the message asks to release the load limit on the current usage point.

2.   Data operator completes the task by contacting the field engineer.

If the trigger date is in the past, Connexo just completes the process meaning that it will only update the corresponding usage point attributes. In this case, no command will be sent to the device or the field engineer won’t be asked to go on site.

3.   Connexo completes the process.

  • Connexo updates the usage point attribute Limiter in the widget Technical information.

  • If a load limit has been defined, then Limiter = Yes. The fields Load limiter type and Load limit become visible and are updated with their corresponding values as they were defined in the process parameters.

  • If no load limit has been defined (Load limit = 0), then Limiter = No.

  • The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

3

Note

During the Limit load (non smart) process, the connection state of the usage point doesn’t change.

Limit load service (out-of-the-box process)

Trigger the Limit load service process to plan the setting of a load limit on smart meters. Connexo will update the corresponding usage point attributes and send commands to the concerned devices.

5

Steps

1.   Start the Limit load service process on a usage point.

The parameters that are specific for this process appear on the Start process page.

Parameter

Description

Load limit

Quantity for the load limit.

Trigger date

Date and time when the process is actually triggered.

Expiration date

Date and time when the load limit should be released.

Reason

Reasons related to electricity can be a lightning strike or a potential overload of the grid. Thermal, gas and water related reasons can be a leak or a pipe burst. Other reasons may be problems on site, security reasons or bad debt.

2.   Click Start to start the process.

 

What happens when the process is running?

5

Steps

1.   Connexo executes the process.

 

If the trigger date is in the past,

  • Connexo updates the usage point attribute Limiter in the widget Technical information.

  • If a load limit has been defined, then Limiter = Yes. The fields Load limiter type and Load limit become visible and are updated with their corresponding values as they were defined in the process parameters.

  • if no load limit has been defined (Load limit = 0), the Limiter = No.

  • The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

 

If the trigger date is in the future and the value in Load limit is higher than 0, then Connexo triggers the target Head End System (HES) to execute the command to enable the corresponding load limit.

 

If the trigger date is in the future and the value in Load limit is 0, then Connexo triggers the target HES to execute the command to release the load limit.

2.   Connexo triggers the target HES to execute the command to enable or disable the load limit.

If there is no response after some pre-defined period, then Connexo stops the flow. Status becomes Aborted and a text variable in Status Overview under the History tab indicates what has happened. The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

If the process was triggered on a usage point with an activated meter that is managed by a HES that isn’t supported, then a new user task is assigned to the data operator. This task contains a message that asks the data operator to contact another operator from the concerned HES to enable or disable the load limit with the defined load limit value for the current usage point on the specified trigger date. As an optional parameter an actual date can be added to this user task. By default, this date is set to the trigger date.

If the process is triggered on a usage point with an activated meter that is managed by a supported HES like Connexo MultiSense , then the HES executes the load limit command.

3.   The HES executes the load limit command.

If the load limit command executes successfully, then Connexo updates the usage point attribute Limiter in the widget Technical information.

  • If a load limit has been defined, then Limiter = Yes. The fields Load limiter type and Load limit become visible and are updated with their corresponding values as they were defined in the process parameters.

  • If no load limit has been defined (Load limit = 0), then Limiter = No.

If the load limit command fails, then Connexo won’t update the usage point attribute Limiter.

The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

4.   Continuation of the process.

If the value in Load limit is higher than 0 and the expiration date is set to a date later than the trigger date, then Connexo waits until the expiration date is achieved. At this moment Connexo re-executes the process with the following values:

  • Load limit = 0.

  • Trigger date = Expiration date from the previous execution.

  • Expiration date is not set.

In all other cases, the process disappears from the tab Running processes and is shown in the History tab on the Processes page.

Link metrology configuration (out-of-the-box process)

Link a metrology configuration to a usage point to define what reading types will be available on a usage point.

5

Steps

1.   Start the Link metrology configuration process on a usage point.

The parameters that are specific for this process appear on the Start process page.

Parameter

Description

Metrology configuration

Select an item from the list of active metrology configuration that are applicable for the current usage point. This field is required.

Link at

Date and time when the metrology configuration becomes linked to the usage point. This field is required.

Purposes

List of required and optional (if any) purposes for the selected metrology configuration. Select any of the needed optional purposes.

Usage point transition

Select an item from the available transitions to the operational stage in the usage point’s life cycle.

2.   Click Start to start the process.

 

What happens when the process is running?

5

Steps

1.   Connexo executes the process.

2.   Connexo checks if the usage point is in a pre-operational or suspended stage.

If it isn’t the case, a user task is created requesting to move the usage point to the appropriate stage.

3.   Connexo tries to link the metrology configuration.

If successfully linked, then the process proceeds to the next step.

If the metrology configuration contains custom attribute sets, then a user task is created asking to link the metrology configuration manually using the ‘Link metrology configuration’ wizard because custom attribute sets can’t be managed in a user task. See “Link a metrology configuration to a usage point”.

If the metrology configuration doesn’t allow gaps but not all needed meters are specified, then a user task is created asking to install the meters first.

If there are errors that are related to incorrect dates and device life cycle stages of linked meters, then the process is aborted and the appropriate error message is shown in the process execution log.

If the linked meters don’t provide reading types for the active purposes of the new metrology configuration, then a user task is created requesting to replace the meters or to try linking another metrology configuration.

4.   Connexo verifies if a transition to an operational stage is specified on the Start process page.

If no appropriate transition was specified, then a user task is created asking to move the usage point to its operational stage on a date that is later than or equal to the link date of the new metrology configuration.

5.   Connexo makes the transition to the operational stage using the specified date.

If the transition is failed, then a user task is created asking to move the usage point to its operational stage on a date that is later than or equal to the link date of the new metrology configuration.

6.   Connexo completes the flow.

  • The metrology configuration is linked to the usage point on the specified link date.

  • On the Metrology configuration widget of the usage point landing page you can see that the new metrology configuration is now linked to the usage point mentioning also the link date.

  • The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

 

Important

The detailed process diagram can be seen in Connexo Flow. If you don’t have sufficient privileges due to a technical or commercial reason, find out who does or contact the Connexo service desk.

If the process is triggered on a usage point with an activated meter that is managed by a supported HES like Connexo MultiSense , then the HES executes the on demand read command.

3.   The HES executes the on demand read command.

If the on demand read command executes successfully, then the process disappears from the tab Running processes and is shown in the History tab on the Processes page.

If the on demand read command fails, then Connexo stops the flow. Status becomes Aborted and a text variable in Status Overview under the History tab indicates what has happened. The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

Remove meter (out-of-the-box process)

Use this process to remove a meter.

5

Steps

1.   Start the Remove meter process on a usage point.

The parameters that are specific for this process appear on the Start process page.

Parameter

Description

Meter name

List of meters that are activated on the usage point. From this list, select the meter that must be removed.

Removal date

Specify date, hour and minutes when the meter must be removed.

2.   Click Start to start the process.

 

What happens when the process is running?

5

Steps

1.   Connexo executes the process.

Connexo verifies if the Allow gaps flag is selected on the linked metrology configuration at removal date.

If gaps aren’t allowed and the metrology configuration is still active after removal date and the meter is required for the calculation of the active purposes of the metrology configuration, then a new user task is created asking to unlink the metrology configuration.

After completing this task the process will be aborted.

If gaps are allowed, then the process proceeds to the next step.

2.   Connexo checks the removal date.

If the removal date is in the past, then Connexo tries to unlink the meter from the usage point at the configured removal date.

If the removal date is in the future, then a new user task is created asking the user to define the time slot for the removal of the meter in the field Actual removal date.

3

Note

The specified actual removal date is taken into account by the process instead of the removal date that was specified when starting the proces.

When the actual removal date has been specified and the user task completed the subprocess Remove meter (physically) starts.

a. A new user task is created asking to remove the meter on the usage point at the configured actual removal date.

b. Connexo schedules a task to read the old meter on the removal date in the target HES.

If the meter isn’t managed by the HES or doesn’t support automatic read, then a new user task is created asking to provide manual meter readings for the meter until the removal date.

c. Connexo waits for a response from the HES. This can be a successful or failed meter reading.

If there was a time-out or if the HES reports a failed command execution, then a new user task is created asking to provide manual meter readings until the removal date.

d. Connexo unlinks the meter.

e. User completes the Remove meter user task from step a.

3.   Connexo completes the process.

  • The old meter is unlinked from the usage point starting from the specified actual removal date.

  • The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

 

1

Replace meter (out-of-the-box process)

Manage the replacement of a meter.

5

Steps

1.   Start the Replace meter process on a usage point.

The parameters that are specific for this process appear on the Start process page.

Parameter

Description

Old meter name

Select one of the meters that are activated on the concerned usage point.

New meter name

Enter the new meter that will replace the existing one.

Replacement date

Enter the date and time when the replacement must take place.

Reason

Enter the reason why the existing meter must be replaced.

2.   Click Start to start the process.

 

What happens when the process is running?

5

Steps

1.   Connexo executes the process.

If the trigger date is in the past

1.   A new user task is defined to a data operator. It asks for the name of the newly installed meter.

If the new meter is already set, then the process immediately proceeds to step 2.

2.   Complete the open user task.

3.   Connexo tries to attach the new meter to the selected usage point.

If the new meter isn’t compatible with the metrology configuration requirements or if the new meter isn’t found in Connexo, then you are asked to enter another meter.

4.   Connexo completes the flow.

  • The new meter is assigned to the usage point and replaces the existing meter.

  • On the usage point landing page you can see that the new meters is now linked and the previous meter is now unlinked at the same date and time.

  • The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

 

If the trigger date is in the future

1.   A new user task is assigned to a data operator. It asks to define the actual date and time for the meter replacement. The Actual replacement date will be taken into account by the process instead of the Replacement date.

2.   Complete the open user task.

The Replace meter (physically) subprocess is started.

3.   A new user task is created for a data operator. It asks to enter the serial number, the device type and the year of certification of the new meter that replaces the existing one.

4.   Connexo schedules a task to read the older meter on the defined replacement date.

If the old meter isn’t managed by the headend system (HES), if the command to the HES fails or if the meter simply doesn’t support automatic reading, then a new user task is created for a data operator asking to provide the meter readings for the old meter until the specified replacement date.

5.   Connexo checks if the metrology configuration linked to the selected usage point allows gaps between meters at the replacement time.

If gaps are allowed, then the old meter is unlinked from the selected usage point on the actual replacement date. Following, the Decommission device process is triggered on the old device in Connexo MultiSense .

Whether or not gaps are allowed is configured on the metrology configuration landing page by the field Allow gaps.

6.   Only in case of a failed automatic meter reading, replace the old meter with the new one and complete the open user task (physical replacement) by providing all the required attributes of the newly installed meter.

7.   Connexo checks if a meter with the same name or serial number as the specified new meter is already installed.

If the meter isn’t’ yet installed in Connexo, then Connexo will try to create the device first. The device configuration of the device type is deduced based on the requirements of the actual metrology configuration on the replacement date. If the appropriate device configuration isn’t found or if the device type isn’t found or in case multiple device configurations are found, then a new user task is created for a data operator asking to manually create a new meter using the required parameters.

8.   When a meter with the name or serial number that has been specified for the new meter has been found, Connexo will verify if the meter can be linked to the selected usage point at the actual replacement date.

If the meter couldn’t be linked to the selected usage point, then a new user task is created for the data operator. It asks to provide a new serial number, device type and year of certification for the new meter.

9.   The Activate device process is triggered on the new device in Connexo MultiSense .

10.   Connexo verifies if the device is in the operational stage and fetches the date when it transitioned to this stage.

If the device isn’t in the operational stage of its life cycle, then the Activate device process is triggered in Connexo MultiSense.

11.   Connexo verifies if the metrology configuration that is linked to the selected usage point at replacement time allows gaps between meters.

12.   If gaps between meters are allowed by the metrology configuration of the selected usage point, then Connexo tries to link the new meter to the usage point using the activation date of the new meter.

If the metrology configuration of the selected usage point doesn’t allow gaps between meters, then the new meter is linked and the old meter is unlinked at the same time and using the activation date of the new meter. In this case, the Decommission device process is triggered on the old device in Connexo MultiSense .

13.   Connexo then continues with the same steps as when the trigger date would have bee in the past, i.e. a new user task is defined to a data operator. It asks for the name of the newly installed meter.

If the new meter is already set, then the process immediately proceeds to step 15.

14.   Complete the open user task.

15.   Connexo tries to attach the new meter to the selected usage point.

16.   If the new meter isn’t compatible with the metrology configuration requirements or if the new meter isn’t found in Connexo, then you are asked to enter another meter.

17.   Connexo completes the flow.

  • The new meter is assigned to the usage point and replaces the existing meter.

  • On the usage point landing page you can see that the new meters is now linked and the previous meter is now unlinked at the same date and time.

  • The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

 

Important

The detailed process diagram can be seen in Connexo Flow. If you don’t have sufficient privileges due to a technical or commercial reason, find out who does or contact the Connexo service desk.

If there is no response after some pre-defined period, then Connexo stops the flow. Status becomes Aborted and a text variable in Status Overview under the History tab indicates what has happened. The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

If the process was triggered on a usage point with an activated meter that is managed by a HES that isn’t supported, then a new user task is assigned to the data operator. This task contains a message that asks the data operator to contact another operator from the concerned HES to restore service for the current usage point on the specified device on the specified trigger date.

If the process is triggered on a usage point with an activated meter that is managed by a supported HES like Connexo MultiSense , then the HES executes the on demand read command.

3.   The HES executes the restore service command.

If the restore service command executes successfully, then the usage point connection status is changed from Logically disconnected to Connected. The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

If the restore service command fails, then Connexo stops the process. Status becomes Aborted and a text variable in Status Overview under the History tab indicates what has happened. The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

Seal (out-of-the-box process)

Trigger the seal process on a usage point to plan its sealing.

5

Steps

1.   Start the Seal process on a usage point.

The parameters that are specific for this process appear on the Start process page.

Parameter

Description

Trigger date

Date and time when the process is actually triggered.

Reason

Reasons related to electricity can be a lightning strike or a potential overload of the grid. Thermal, gas and water related reasons can be a leak or a pipe burst. Other reasons may be problems on site, security reasons or bad debt.

2.   Click Start to start the process.

 

What happens when the process is running?

5

Steps

1.   Connexo executes the process.

If the trigger date is in the future, then a new user task is created and assigned to a data operator. This task contains a message that asks the data operator to contact a field engineer to seal the current usage point on the specified trigger date. As an optional parameter an actual date can be added to this user task. By default, this date is set to the trigger date.

2.   The data operator completes the user task by contacting the field engineer.

3.   Connexo completes the flow.

  • Connexo updates the usage point’s connection status from Connected to Physically disconnected.

  • The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

Unlink metrology configuration (out-of-the-box process)

Unlink a metrology configuration from a usage point to support the ‘move out scenario’. After unlinking you will be able to link another metrology configuration to the usage point. See “Link metrology configuration (out-of-the-box process)”.

5

Steps

1.   Start the Unlink metrology configuration process on a usage point.

The parameters that are specific for this process appear on the Start process page.

Parameter

Description

Unlink at

Specify when to unlink the metrology configuration from the current usage point. This is a required field.

Usage point transition

Select a transition from the list of transitions that are available to move from the current state to a state in the suspended stage in the usage point’s life cycle.

2.   Click Start to start the process.

 

What happens when the process is running?

5

Steps

1.   Connexo executes the process.

2.   Connexo verifies if the metrology configuration that must be unlinked is active at the specified date.

If the metrology configuration is active, then the process continues to the next step.

If no metrology configuration is found or a metrology configuration is found with a ‘stop’ date before the specified unlinking date, then the process is aborted.

3.   Connexo verifies if the usage point is in an operational state on the specified unlink date and later.

If the usage point isn’t in an operational state, then the metrology configuration is unlinked. See step

4.   Connexo schedules the transition of the usage point on the specified date.

If a transition wasn’t specified on the Start process form, then the transition fails. A user task is created requesting to transition the usage point manually from its operational stage.

5.   User completes user task.

Connexo waits until the usage point’s state has been changed to reverify if the usage point is still in an operational state. See step 3. 

6.   When the specified unlink date has been reached and if the usage point is in the suspended stage, then Connexo unlinks the metrology configuration from the usage point.

7.   Connexo completes the flow.

  • The metrology configuration is shown on the usage point’s History page with a start and end date in the column Period. See “View history of linked metrology configurations”.

  • The custom attribute sets that are related to the unlinked metrology configuration are still visible on the Usage point attributes page and will remain to be visible during the entire lifetime of the usage point.

  • On the Metrology configuration widget of the usage point landing page you can see that the metrology configuration is now unlinked from the usage point.

  • The process disappears from the tab Running processes and is shown in the History tab on the Processes page.

If unlinking the metrology configuration fails, then the process is aborted.
